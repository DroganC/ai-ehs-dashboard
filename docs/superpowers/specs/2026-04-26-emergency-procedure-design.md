# 应急流程卡片（EMERGENCY_PROCEDURE）设计规格

**日期:** 2026-04-26  
**工程:** `triple-slot-h5`  
**状态:** 已采纳（含 MobX、2 关、GameCode 定名）

## 1. 目标与成功标准

- **目标:** 知识巩固，帮助用户记熟**标准应急步骤顺序**；弱考核、可反复重玩。  
- **成功标准:** 玩家在无持久化、无干扰项的情况下，能按关卡给出的步骤池完成**正确排序**；放错有明确反馈；轻提示不泄露「下一张该放哪张牌」的具体答案。

## 2. 范围外（首版明确不做）

- 干扰牌（非本流程步骤的牌）  
- 强提示、分数/星级/排行榜、解锁逻辑  
- 本地存储与后台进度同步  
- 关卡数超过 **2 关** 以外的更多关卡（可预留扩展点，但首版只交付 2 关数据）

## 3. 产品信息


| 项        | 值                                                 |
| -------- | ------------------------------------------------- |
| 代码/枚举    | `EMERGENCY_PROCEDURE`（`GameCode`）                 |
| 列表/顶栏展示名 | **应急流程卡片**（与 UI 图一致，可与代码名并存）                      |
| 关卡       | **固定 2 关**（`level-01.json`、`level-02.json` 或等效命名） |


## 4. 用户流程（不持久化 + 按序闯关）

- 从首页/游戏列表进入本游戏。  
- **总是从第 1 关开始**；仅在本会话/本局内通过「下一关」进入第 2 关。  
- **刷新页面或离开再进**：不保留进度，**再次从第 1 关开始**。  
- 第 2 关通关后：展示全通/结束态（文案可配），再玩可从第 1 关重开或返回列表（行为与现有 H5 导航一致即可，首版不强制后端）。

## 5. 关卡与数据结构

- 关卡为 **JSON**，放在 `public` 下按游戏子目录管理（与 `triple-slot` 的 `public/games/triple-slot/levels/` 模式对齐），例如：  
`public/games/emergency-procedure/levels/level-01.json`、`level-02.json`。
- **槽位与预填**由数据驱动：每关的槽位数、哪些槽预填、预填的牌，均在 JSON 中描述。  
- **待选区**只包含本关**正确流程**中尚未被预填占用的牌；开局对这部分牌**洗牌**。  
- **无干扰项**：池中牌集合 = 本关正确流程步骤减去预填步骤。

### 5.1 建议 JSON 字段（实现时可微调命名，但语义应保留）

- `id` / `levelIndex` / `title`：关卡元信息。  
- `cards[]`：`id`、`label`（如「断电」「报警」等展示文）。  
- `slotCount` 或 `slots[]`：描述各槽下标、是否 `prefillCardId`（预填时指向 `cards` 的 `id`）。  
- `correctOrder: string[]`：与**从左到右/从上到下**槽位一一对应，元素为 `card id`，与预填内容一致。  
- 可选：`ui`（标题、底部黄字提示、完成/全通文案）、`hintEnabled`（本关是否开启轻提示）。

**校验规则:** 玩家将牌放入某槽时，若 `cardId !== correctOrder[slotIndex]`，则**该牌回到待选区**（可伴随轻量 toast/动效），不清空已正确步骤。

**通关条件:** 所有非预填槽位已放置且与 `correctOrder` 一致，或等价的「全槽填满且全对」。

## 6. 轻提示（B）

- 不直接展示「下一张应放置的牌名」。  
- **实现建议:** 对「当前**顺序**上第一个空槽位」做轻量高亮（如边框/微光/序号强调），引导玩家**按位序**填充；若某关需要关闭提示，用数据开关关闭。

## 7. 状态管理（MobX）

- 使用 **MobX** 管理可观察状态与动作（`mobx` / `mobx-react-lite` 已在项目依赖中）。  
- 建议单 stores 模块，例如 `emergencyProcedureStore` 或同游戏前缀，包含但不限于：  
  - 当前关卡下标、加载中的关卡数据；  
  - 各槽位当前牌、池中剩余牌（已洗牌）；  
  - 本关是否完成、是否可进入「下一关」、是否**全游通关**；
- View 以 `observer` 包裹，与现有 H5 技术栈（React 19、Vite）一致。

## 8. 注册与路由

- 在 `GameCode` 中增加 `EMERGENCY_PROCEDURE`。  
- 在 `src/games/registry` 中注册：标题/描述/组件，**不**在其它游戏模块间交叉 import。  
- 本游戏与「三消槽位」等并列出现在游戏列表/路由中，遵循现有 `routes` 与 `GamePlayPage` 约定。

## 9. 界面与动效

- 视觉**参考**用户提供的线框/截图：头区（标题、待放置张数）、中部待选/卡片区、底部槽位条、底部说明黄字。  
- 不强制 1:1 像素，但**版式、圆角、暖色大地色系**与信息层级应一致；具体 Less 文件放在本游戏子目录。  
- 放错**归位**需有过渡，避免硬跳变。

## 10. 测试建议（手测）

- 两关各完成一次、中间「下一关」、刷新后从第 1 关。  
- 变槽位/预填组合：用 level-01 / level-02 各覆盖不同槽数或预填位置（数据即测试用例）。  
- 轻提示开/关（若第 2 关关闭提示也可测一次）。
